This week we continued the discussion about getters and setters and how they relate to Object-Oriented design patterns. In our second reading about getters and setters, the author dileneated between getters/setters in business logic and in User Interfaces. The principle is about keeping business logic free from them so that implementation details are encapsulated.

The article mentioned using a Builder pattern for connecting business logic and application state to user interfaces. The benefit of using a Builder is that the same application state can be used to build user-interfaces to multiple output environments without being tied to specific implementation details.

The article's sample code was too verbose for me to endorse as a viable approach, but the idea of having Builders that generate user-facing outputs from a single source-of-truth is something that I resonate with. The main issue with the article's approach (in my view) is their usage of Java. To demonstrate, I'll show a side-by-side comparison of the sample Java code with an equivalent approach using JavaScript and template literals.

While they both accomplish the same result, the Java version is much more prone to developer error, as the strings are difficult to read, and the hierarchy of the html is obfuscated. The beauty of javascript template literals is that they provide the ergonomics of a templating engine without a DSL, any logic within the interpolations is just javascript.

In today's world, there are more devices, services, and environments than there ever have been, and each has a set of constraints that define the proper 'files' to build.

By setting up the codebase to separate business logic from UI builders, we can deploy to multiple target environments with ease.

For example, it's common to build websites with public information (like documentation) with two versions: one for the web using HTML/CSS/JS for a human experience, and one for LLMs using markdown. You can try this on Bun docs. By appending a .md to any documentation page, you get a markdown file that contains the same inforamtion, but formatted for LLMs. It makes sense to have 'buidlers' for each output format, without having to worry about the implementations of the internal state of the source-of-truth during doc generation.

Pick of the week

vhs

VHS is a CLI recording program that allows you to define a .tape that includes a deterministic set of events to execute in a CLI session.

VHS will run that file and generate an output animation, which can be very helpful for authoring tutorials or recording workflows.
