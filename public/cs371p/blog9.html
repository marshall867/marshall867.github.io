WEEK #9

This week we learned about initializer lists in C++. I used to be confused about the various ways to initialize variables, but learning about initializer list constructors cleared some things up. The fact that there are so many ways to define constructors for a single object is interesting to me. Other languages I'm familiar with usually just have one way to instantiate an object defined by a class. In C++, it seems normal to have four or five different types of constructors!

By this point, we've re-derived the need for many of the items in the standard library. Learning about the mechanics of std::array, std::vector, and std::allocator has deepened my understand of what these seemingly simple helpers do.

By implementing our own Allocator for project #3, I was exposed to the various considerations when allocating memory. We used a simple strategy of "first-fit" and integer sentinels, but I'm sure there are many ways in which this type of data structure could be optimized for certain workloads and data types.

I was a little surprised at how verbose C++ classes end up being, even when they expose a simple API. While creating my Allocator, having to implemement both a read/write iterator and a read-only iterator required either repeating code or using complex casting statements. The result is that the important logic of the file gets hidden among lines and lines of boilerplate.

I tend to prefer vertically compact code, such that most of the visual real estate is logically impactful code. That way, "simple" components are short files, and "complex" classes are 200-500 lines. Any longer, and I find that I constantly have to jump back and forth between vertical sections of the code.

I'd like to learn more about how to structure C++ classes in a way that minimizes verbosity. I've gained a lot of respect for the power and compatibility of C++, but I'd love how to keep any necessary complexity as hidden as possible.

Pick of the week

Helix Editor

I've been trying out different code editors this semester. I used to rely on VSCode for everything, but I wanted to see if I could switch to a terminal-based editor. Working from the terminal is daunting at first, but it had too many benefits for me to ignore any longer:

  • Mouse-free navigation
  • Machine-agnostic workflow
  • Looks cooler

Until now, I've been using neovim as my primary editor. But, for the past week, I've been editing with Helix.

Why Helix?

  • Native support for multiple cursors
  • Better tree-sitter motions
  • Selection -> Verb instead of Verb -> Selection

Running hx --tutor is a nice way to learn the most common keyboard shortcuts and commands if you're interested in trying it.
