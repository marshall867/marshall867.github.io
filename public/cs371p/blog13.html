WEEK #13

This week we learned about the role of virtual functions and abstract class hierarchies. Specifically, we learned about pure virtual methods and pure virtual destructors.

Pure virtual methods allow a class to define methods that need to be implemented by children without providing a default behavior. Methods are defined as "pure virtual" by declaring "virtual void someMethod() = 0;". This makes that class "abstract", which means that you cannot create objects of that class directly. The only way to instantiate these objects is to use a subclass that implements the required methods.

When the parent class has default implementations for all of its methods, it's common practice to declare a virtual destructor "virtual ~AbstractParent() = 0;". This pure virtual destructor is sufficient for making a class abstract in C++.

Abstract classes can still hold state and provide methods that can be used by child classes. As we learned in class, when considering "Shape" and "Circle", certain data properties fit well in the parent abstract class, like position x,y. For reading shape parameters from std::cin, it's useful to have "Shape::read(istream)" that "Circle::read(istream)" can call.

Professor Downing mentioned a story from his work experience where an upstream library author changed the function signature of a specific method, adding a "const" annotation to a method, asserting that the method didn't mutate the state of the instance. This change caused the downstream child class to no longer override the library properly - without causing a compile-time error.

The solution to this is to use the override keyword, which will cause a compiler error when it doesn't match the function signature of a parent method.

C++ has many intricacies, and understanding the syntax and language features is important for interacting with real-world codebases. Simpler languages hide the complexity, which makes it easier to get started, but "magic" usually conceals internal design decisions with tradeoffs. C++ makes everything explicit, which requries more effort and reduces readability, but it gives developers full control over their code.

Pick of the week

toybox

Toybox is a single executable that includes the most common linux CLI utilities, like ls, cat, mkdir, whoami, chmod, etc.

This is especially useful for resource-constrained or simplified systems where you want the simplicity of posix commands you already know, without having to manually implement or import each individual command.

I heard about this tool from Operating Systems, and I appreciated the fact that well-known standards for program behaviors can be implemented in many different ways, as long as they fulfil the user-contract (return same output for same input). This is similar to the idea of child classes that implement abstract class methods.
